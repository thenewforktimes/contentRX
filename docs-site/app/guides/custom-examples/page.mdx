# Custom examples

Teams ship product copy with intentional voice quirks — phrasings
that intentionally read differently from the defaults. ContentRX's
default rules sometimes flag these. **Custom examples** are the
surgical alternative to globally disabling a rule: mark the specific
string as correct (or known-bad) for your team, and ContentRX
short-circuits the LLM on every subsequent evaluation of it.

One example usually makes it land:

> Your team ships `Let's go.` on confirmation banners. ContentRX's
> default mechanics rules flag trailing periods on heading-style
> microcopy, which catches this. Disabling the rule globally would
> lose its protection elsewhere. Adding `Let's go.` as a custom
> example (verdict = `pass`) means ContentRX stops questioning that
> specific string without touching the rule anywhere else.

## Ingestion lives in MCP and the CLI

Custom examples are managed from the surfaces where your team
already authors copy — Claude Code, Cursor, your terminal — not from
a web form. The content decision happens during authoring, so the
"save this" action happens there too.

### From Claude Code or Cursor (MCP)

The `contentrx-mcp` server exposes four tools:

- `custom_example_add` — mark a string as correct or known-bad
- `custom_example_list` — list the team's current set
- `custom_example_search` — check whether a string is covered
- `custom_example_remove` — delete an entry

In practice:

> **You:** We always ship `Let's go.` on confirmation banners. Save
> that as a custom example — it's intentional voice.
>
> **Claude:** *(calls `custom_example_search("Let's go.")` — not
> covered yet, then `custom_example_add` with
> `verdict=pass`, `moment=confirmation`, and a short note)*
>
> *Added. ContentRX will short-circuit on that string going
> forward.*

### From the terminal (CLI)

The `contentrx` CLI exposes the same operations:

```bash
contentrx example add "Let's go." \
  --verdict pass \
  --moment confirmation \
  --notes "Intentional conversational voice on confirmations"

contentrx example list
contentrx example search "Let's go."
contentrx example remove <id>
```

Two helpers for bulk work:

```bash
contentrx example export > custom-examples.json   # backup / check into git
contentrx example import custom-examples.json     # restore / seed a new team
```

Import skips duplicates by default. Pass `--strict` to fail on them.

## Anatomy of an entry

Every entry carries:

| Field | Required | Notes |
|---|---|---|
| `text` | Required | The exact string to short-circuit on. Case + whitespace are normalised at match time, but punctuation is not — `Let's go.` and `Let's go` are distinct entries. |
| `verdict` | Required | `pass` or `violation`. Use `pass` for "this string is correct for us"; `violation` for "this string is always wrong for us." |
| `moment` | Optional | Scope the match to one context (e.g., `confirmation`). When omitted, the entry matches in any context. |
| `content_type` | Optional | Same idea, scoped to a content type (e.g., `button_cta`). |
| `notes` | Recommended | 1–3 sentences explaining why the entry exists. Surfaced on match so teammates see why the verdict is what it is. |

## What happens when a match fires

ContentRX's `/api/check` finds the match before calling the LLM.
The evaluation returns the stored verdict immediately:

- No LLM tokens spent on the match.
- Your monthly scan quota still decrements by one (a scan is a scan).
- The Figma plugin shows a small "Team custom example" tag on the
  card. Display-only; no new CTA — the team already decided.

## Limits and guardrails

- **500 entries per team**, enforced at create time.
- **Immutable after creation.** The `text`, `verdict`, `moment`, and
  `content_type` can't be edited in place. Only `notes` is updatable.
  To change what an entry matches, delete it and add the replacement
  — the audit trail stays clean.
- **Unique per team.** The same normalised text can't exist twice in
  one team's set. Prevents silent conflicts where two entries
  disagree on the verdict.

## Privacy

`text` is plaintext, team-authored, team-owned. Scoped by the team
owner's user id. Nothing in a custom example is derived from a
user's input — it's a decision your team made and published to the
tool.
